[&>a]:underline-offset-2 [&>a:hover]:underline [&_a]:text-accent-blue [&_a:hover]:underline d:T3526,<h2>The Dashboard Graveyard Problem</h2>
<p>Every IT organisation has a dashboard graveyard. Reports that took weeks to build, that someone looked at three times, and that now sit forgotten in a SharePoint folder.</p>
<div style="margin: 1.5rem 0; border-radius: 12px; overflow: hidden; border: 1px solid rgba(255,255,255,0.08);">
<img src="https://images.unsplash.com/photo-1551288049-bebda4e38f71?w=800&q=80&auto=format&fit=crop" alt="Data analytics and PowerBI dashboard charts" style="width:100%; height:280px; object-fit:cover; display:block;"/>
<p style="background:#0f172a; color:#475569; font-size:0.72rem; padding:6px 12px; margin:0; text-align:right;">Photo: Unsplash</p>
</div>
<p>I built my first IT dashboard in 2018. By 2020, it was in the graveyard. I rebuilt it in 2022 with a different approach — this time anchored in the questions IT leadership actually asked, not the metrics I thought they should care about.</p>
<p>The second dashboard gets reviewed every Monday.</p>
<h2>Start With the Question, Not the Data</h2>
<p>The most common dashboard mistake: building from available data outward. You have SNOW data, you have asset data, you have AD data — so you build charts from all of it.</p>
<p><strong>The right approach:</strong> Start by asking every person who will use the dashboard: <em>"What questions do you need answered each week to do your job?"</em></p>
<p>At a large enterprise, the answers were:</p>
<ol>
<li>Senior Leadership: "Are we meeting our SLAs? Any big issues this week?"</li>
<li>IT Manager: "Where are tickets backing up? Which engineer is overloaded?"</li>
<li>Finance (quarterly): "What are we spending on IT? What's the asset refresh bill?"</li>
<li>IT Team: "What's open and due today?"</li>
</ol>
<p>Four users, four different dashboards — or four pages in one dashboard with the right navigation.</p>

<h2>Connecting PowerBI to Real IT Data Sources</h2>
<h3>ServiceNow → PowerBI</h3>
<p>ServiceNow has a REST API that PowerBI can connect to directly:</p>
<pre><code>Source: REST API
URL: https://yourinstance.service-now.com/api/now/table/incident
Authentication: Basic (service account)
Parameters: ?sysparm_query=assigned_to=&#x3C;your_group>&#x26;sysparm_fields=number,state,priority,opened_at,resolved_at
</code></pre>
<p>Set refresh to every hour for operational dashboards. Daily for trend reports.</p>
<h3>SharePoint Lists → PowerBI</h3>
<p>For IT asset data maintained in SharePoint:</p>
<ol>
<li>Get Data → SharePoint Online List</li>
<li>Authenticate with M365 credentials</li>
<li>Transform: filter out system columns, parse dates, create calculated columns</li>
</ol>
<p>SharePoint lists are ideal for small-to-medium asset registers (&#x3C; 50,000 rows) — the query performance is excellent.</p>
<h3>Enterprise ERP Systems → PowerBI</h3>
<p>For SAP or other ERP environments, expose metrics via:</p>
<ul>
<li>OData endpoint if available (preferred)</li>
<li>SharePoint export if OData isn't configured</li>
<li>CSV export as last resort (loses real-time capability)</li>
</ul>
<h2>The 5 Charts Every IT Operations Dashboard Needs</h2>
<h3>1. SLA Compliance Rate (Line Chart, Rolling 30 Days)</h3>
<p>Shows trend, not just current state. Everyone can see if it's improving or degrading.</p>
<h3>2. Ticket Volume by Category (Stacked Bar, Monthly)</h3>
<p>Reveals patterns — seasonal spikes, category growth, the impact of KB articles on volume.</p>
<h3>3. Mean Time to Resolution by Priority (Card + Gauge)</h3>
<p>The first question from leadership: "How fast do we fix things?" This answers it clearly.</p>
<h3>4. Asset Warranty Expiry Timeline (Timeline/Calendar)</h3>
<p>Shows the next 12 months of warranty expirations. Prevents budget surprises. Colour-code: green (>12 months), amber (6–12 months), red (&#x3C;6 months).</p>
<h3>5. Open Tickets Aging (Heat Map or Matrix)</h3>
<p>Which tickets are oldest? By engineer? By category? This one prevents SLA breaches by making aged tickets visible before they breach.</p>

<h2>Getting Buy-In: How to Make People Actually Use Your Dashboard</h2>
<p>The best dashboard in the world fails if people don't open it. Adoption tactics that worked for me:</p>
<ol>
<li><strong>Include it in Monday's IT meeting</strong> — pull it up on screen, walk through it. Creates the habit.</li>
<li><strong>Email the executive summary page every Monday morning</strong> — one screenshot, three bullet points, no login required.</li>
<li><strong>Ask for feedback in week 2</strong> — "Is there anything you're not finding on here?" People will tell you exactly what they need.</li>
<li><strong>Name it after the audience</strong> — "IT Operations Dashboard" is for IT. "IT Service Report — Leadership View" is for senior stakeholders. Different names signal different purposes.</li>
</ol>
<p>The goal is to become <em>the source of truth</em> for IT metrics. Once your dashboard is the thing people reference in meetings, it will always be maintained because there's a clear cost to letting it go stale.</p>
