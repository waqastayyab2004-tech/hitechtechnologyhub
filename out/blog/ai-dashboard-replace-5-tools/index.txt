<h1>How I Replaced 5 Daily IT Tools With One AI Dashboard — Built From Scratch</h1>
<p>Every morning started the same way. Open SAP Outlook for emails. Switch to ServiceNow to check ticket assignments. Open Gmail for personal mail. Jump to OneDrive for files. Run a terminal command to check SLA status. Repeat all day.</p>
<p>Five tools. Constant context-switching. And the worst part? I was still missing things — a P1 ticket assigned at 2 PM, an SLA approaching breach at 4 PM, an urgent email buried in a 200-message inbox.</p>
<p>So I built one app that does everything. Here's how.</p>

<h2>The Problem: Too Many Tabs, Too Many Misses</h2>
<p>As a Senior IT System Engineer managing 200+ users across three offices in Riyadh, my daily workflow depended entirely on remembering to check the right browser tab at the right moment. That's not a system — that's hope.</p>
<p>The specific pain points:</p>
<ul>
<li><strong>SLA breaches</strong> were reactive. I only found out when a manager escalated. By then, the ticket had already breached P2 response time.</li>
<li><strong>New ticket assignments</strong> required me to keep ServiceNow open and refresh it manually. Miss a refresh, miss a critical P1.</li>
<li><strong>Email action items</strong> were buried. Reading 40–80 emails a day to extract 3–4 items that actually needed action was a 20-minute daily tax.</li>
<li><strong>Tool fatigue</strong> was real. 5 apps × context-switching overhead = lost focus every 20 minutes.</li>
</ul>
<p>The solution wasn't to work harder. It was to build something smarter.</p>
<hr>
<h2>The Solution: Waqas AI Hub</h2>
<p>The idea was simple: one app that surfaces everything I need to know, proactively, without me asking.</p>
<p><strong>What I built:</strong></p>
<ul>
<li>A Python FastAPI backend with 15+ REST API endpoints</li>

<h2>The Technical Challenges (And How I Solved Them)</h2>
<h3>Challenge 1: SAP Conditional Access Blocks OAuth on Non-Managed Macs</h3>
<p>SAP's Conditional Access policy blocks the device-code OAuth flow on any Mac not enrolled in Intune. That's the standard way to authenticate Microsoft Graph API on a personal device — and it simply doesn't work.</p>
<p><strong>Solution:</strong> Register a <code>localhost:8080</code> redirect URI in Azure AD. Use the authorization code flow instead of device code. When the user authenticates, the browser redirects back to <code>localhost:8080/callback</code>, where FastAPI catches the code and exchanges it for tokens.</p>
<p>This works because <code>localhost</code> is explicitly permitted as a redirect URI for public clients, even on non-managed devices.</p>
<pre><code class="language-python"># FastAPI OAuth callback handler
@app.get("/callback")
async def oauth_callback(code: str):
    token = exchange_code_for_token(code, redirect_uri="http://localhost:8080/callback")
    save_token(token)
    return RedirectResponse("/")
</code></pre>
<h3>Challenge 2: The macOS <code>.app</code> That Wouldn't Launch</h3>
<p>My first attempt at packaging the dashboard as a macOS app was a shell script wrapper. It opened a terminal briefly, then nothing — the app silently died.</p>
<p>The problem: macOS kills child processes when the parent shell script exits. The FastAPI server was the child. Shell script exits → server dies → WebView has nothing to connect to.</p>
<p><strong>Solution:</strong> Rewrite the app in Swift using <code>WKWebView</code>. The Swift process stays alive independently, manages the Python server as a subprocess with <code>nohup</code> and <code>disown</code>, and presents the web UI in a proper native window.</p>
<pre><code class="language-swift">// Launch Python server from Swift
let process = Process()
process.executableURL = URL(fileURLWithPath: "/usr/bin/python3")
process.arguments = [serverPath]
try process.run()
// Process continues running — not killed when Swift launches
</code></pre>
<h3>Challenge 3: ServiceNow Authentication</h3>
<p>ServiceNow at enterprise level doesn't offer a simple API key. Session cookies from Safari DevTools are the pragmatic solution — paste them once, they last ~24 hours.</p>
<p>The dashboard includes a "SNOW Connect" panel where I paste fresh cookies. A Python utility stores them and injects them into every ServiceNow REST API request.</p>

<h2>The WhatsApp Alert System</h2>
<p>This is the feature that actually changed my workday.</p>
<p>Every 5 minutes, a Python daemon:</p>
<ol>
<li>Polls ServiceNow REST API for tickets assigned to my queue</li>
<li>Calculates time-to-SLA-breach for each ticket</li>
<li>If any ticket is within 30 minutes of breach — sends a WhatsApp message</li>
</ol>
<pre><code class="language-python"># SLA breach alert logic
for ticket in my_tickets:
    minutes_remaining = calculate_sla_remaining(ticket)
    if minutes_remaining &#x3C;= 30:
        send_whatsapp(
            f"⚠️ SLA ALERT: {ticket['number']} - {ticket['short_description']}\n"
            f"Priority: {ticket['priority']} | {minutes_remaining} min remaining"
        )
</code></pre>

<h2>What I'd Do Differently</h2>
<ol>
<li><strong>Build the Swift wrapper first, not last.</strong> The shell script <code>.app</code> cost me two days of debugging. Start native.</li>
<li><strong>Design the JSON pipeline from day one.</strong> Early versions sent email notifications from the agents. Switching to JSON output and displaying results as dashboard cards was a much better architecture — but required rewriting all 5 scripts.</li>
<li><strong>Auto-refresh ServiceNow cookies.</strong> The current 24-hour manual paste is the only manual step left in the entire system. A Playwright automation that refreshes cookies on a schedule would complete the automation.</li>
</ol>
