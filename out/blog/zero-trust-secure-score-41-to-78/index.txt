<h1>Zero Trust in Practice: How We Took Microsoft Secure Score from 41% to 78%</h1>
<p>When I started this project, our Microsoft Secure Score was 41%. Not catastrophically bad, but well below where a multinational enterprise with sensitive data and executive users should be.</p>
<p>Twelve months later: 78%. Zero security incidents attributed to identity compromise. No legacy authentication attacks succeeding. All admin access governed through PIM with full audit trails.</p>
<p>Here's exactly what we did — and the order we did it in.</p>

<h2>What Is Microsoft Secure Score?</h2>
<p>Secure Score is Microsoft's measurement of your security posture across Microsoft 365 and Azure AD. It scores you on hundreds of control points and gives you a prioritised list of improvements with estimated score impact.</p>
<p>Think of it less as a report card and more as a backlog. Every item is something you can actually fix.</p>
<p>Starting at 41% means roughly half of the recommended controls weren't in place. Some were complex, some were a single toggle. We started with the toggles.</p>
<hr>
<h2>Phase 1: The Quick Wins (Weeks 1–4)</h2>
<p>These are changes that take under 30 minutes to implement and give significant Secure Score points. Start here.</p>
<h3>1. Block Legacy Authentication</h3>
<p>Legacy authentication protocols (Basic Auth, SMTP AUTH, POP3, IMAP without modern auth) bypass MFA entirely. According to Microsoft, over 99% of password spray attacks use legacy authentication.</p>
<p><strong>Action:</strong> Create a Conditional Access policy that blocks all legacy authentication across all users and all cloud apps.</p>
<pre><code>Policy: Block Legacy Authentication
- Assignments: All users
- Cloud apps: All cloud apps
- Conditions: Client apps → Legacy authentication clients (check all)
- Grant: Block access
</code></pre>
<p><strong>Caveat:</strong> Run in report-only mode for 2 weeks first. Find any printers, scanners, or old applications using SMTP AUTH. Migrate them before enforcing.</p>
<p><strong>Score impact:</strong> +8 to +12 points depending on tenant size.</p>
<h3>2. Require MFA for All Users</h3>
<p>The single highest-ROI security control in Microsoft 365.</p>
<pre><code>Policy: Require MFA — All Users
- Assignments: All users (exclude break-glass accounts)
- Cloud apps: All cloud apps
- Grant: Require multi-factor authentication
</code></pre>
<p>Add a named location for your office network IP ranges and exclude it from the MFA requirement for users already on a trusted corporate network. This reduces friction without reducing security for external access.</p>
<p><strong>Score impact:</strong> +10 to +15 points.</p>
<h3>3. Enable Self-Service Password Reset (SSPR)</h3>
<p>Reduce helpdesk load and eliminate a social engineering attack vector (calling IT to reset passwords).</p>
<p>Require at least 2 authentication methods for SSPR registration. Enable the combined MFA + SSPR registration experience so users complete both in one session.</p>
<hr>
<h2>Phase 2: Device Compliance (Weeks 4–8)</h2>
<h3>4. Require Compliant Device for Corporate Resource Access</h3>
<p>Once MFA is in place, add device compliance as a second condition.</p>
<pre><code>Policy: Require Compliant Device
- Assignments: All users
- Cloud apps: All cloud apps
- Conditions: Device platforms → Windows, macOS, iOS, Android
- Grant: Require device to be marked as compliant
  (require one of the selected controls)
</code></pre>
<p>A non-compliant device triggers a Conditional Access failure — the user sees a page telling them to enrol in Intune or fix their compliance issues.</p>
<p>This is where Intune compliance policies matter. Define what "compliant" means:</p>
<ul>
<li>Windows 11 with BitLocker enabled</li>
<li>Microsoft Defender running with current signatures</li>
<li>No jailbreak or root detection (iOS/Android)</li>
<li>Minimum OS version enforced</li>
</ul>
<p><strong>The result:</strong> A compromised credential is worthless without a compliant device. Credential theft attacks stop at the Conditional Access gate.</p>
<h3>5. BitLocker Enforcement</h3>
<p>BitLocker (XTS-AES 256-bit) on all Windows endpoints with recovery keys escrowed to Azure AD.</p>
<pre><code class="language-powershell"># Intune compliance policy — require BitLocker
Set-MgDeviceManagementDeviceCompliancePolicy -BitLockerEnabled $true
</code></pre>
<p>Recovery keys in Azure AD Admin Center → Devices → [device] → Recovery Keys. Never stored locally, never emailed.</p>
<p><strong>Rollout approach:</strong> 3-month phased enforcement. Start with report-only compliance status, no blocking. This surfaces non-compliant devices without disrupting anyone. Move to enforcement after the 3-month window closes.</p>
<hr>
<h2>Phase 3: Privileged Access (Weeks 8–12)</h2>
<h3>6. Eliminate Permanent Global Admin Accounts</h3>
<p>Permanent Global Admin access is a standing invitation for privilege abuse. Every privileged action should be time-bounded and approved.</p>
<p><strong>Implement Privileged Identity Management (PIM):</strong></p>
<ol>
<li>Identify all permanent Global Admin, Exchange Admin, and Intune Admin role assignments</li>
<li>Convert them to eligible assignments in PIM</li>
<li>Set activation requirements: MFA + justification + maximum 4-hour duration</li>
<li>Configure approval for Global Admin (IT Manager approval required)</li>
<li>Enable alerts for admin role assignments outside PIM</li>
</ol>
<pre><code>PIM Configuration:
- Global Admin: Eligible, requires approval, max 4 hours, MFA required
- Exchange Admin: Eligible, no approval, max 8 hours, MFA required
- Intune Admin: Eligible, no approval, max 8 hours, MFA required
</code></pre>
<p><strong>Score impact:</strong> +15 to +20 points. One of the highest-impact individual controls.</p>
<p>The audit trail alone is worth it — every privileged action is logged with who requested it, who approved it, and when it expired.</p>
<hr>
<h2>Phase 4: Endpoint Security (Weeks 12–24)</h2>
<h3>7. Microsoft Defender for Endpoint</h3>
<p>Enable EDR (Endpoint Detection and Response) across all managed devices. In Microsoft 365, Defender for Endpoint integrates directly with Intune for device risk scoring.</p>
<p>Configure a Conditional Access policy that blocks sign-in from devices with high Defender risk scores:</p>
<pre><code>Policy: Block High-Risk Devices
- Conditions: Device filters → DeviceRiskScore = High
- Grant: Block access
</code></pre>
<p>This creates a feedback loop: Defender detects a threat on a device → device gets a high risk score → Conditional Access blocks all corporate resource access → forces IT investigation before access is restored.</p>
<h3>8. CyberArk EPAM: Remove Local Admin Without Removing Productivity</h3>
<p>Removing local admin from all endpoint users is the right security decision. It's also the decision most likely to generate helpdesk tickets.</p>
<p>CyberArk Endpoint Privilege Manager solves this gracefully. Instead of removing admin entirely, users get a "Privileges" app. When they need to install something, they request elevation through the app. The request is logged, time-limited (15 minutes), and auditable.</p>
<p>Users accept this because the friction is low — one click, a 15-minute window. IT accepts it because every elevation is logged with a business justification.</p>
<hr>
<h2>The DLP Layer</h2>
<p>Data Loss Prevention policies are Secure Score's most underrated improvement category.</p>
<p>Start with three policies:</p>
<ol>
<li><strong>Credit card and financial data</strong> — block external email containing card numbers</li>
<li><strong>Sensitive keyword policy</strong> — flag emails containing terms like "confidential," "board," "salary" going to external recipients</li>
<li><strong>SharePoint external sharing</strong> — block sharing with anyone outside approved domains</li>
</ol>
<p>The first few weeks will surface false positives. Tune the policies with exceptions before moving to enforcement mode.</p>

<h2>The Most Important Lesson</h2>
<p><strong>Deploy in report-only mode first. Always.</strong></p>

<p>One week in report-only mode for every blocking policy. Non-negotiable.</p>
